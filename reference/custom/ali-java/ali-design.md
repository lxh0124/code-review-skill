## 强制（评审门禁）

- 存储方案与底层数据结构须评审通过并文档化；表结构/字段/索引变更须 double check 后上线
- User >1 类且 UC >5：须用例图
- 业务状态 >3 个：须状态图 + 明确转换条件
- 调用链对象 >3 个：须时序图 + 各环节输入输出
- 模型类 >5 个且依赖复杂：须类图
- 对象协作 >2 且流程复杂：须活动图

## 推荐

- 架构设计明确：系统边界、模块依赖、演进原则、非功能性需求（安全/可用/扩展）
- 评估异常流程与业务边界（如支付扣款成功但入账失败）
- 单一职责；优先组合/聚合而非继承（继承须符合里氏替换）
- 依赖倒置：依赖抽象；开闭原则；抽取公共模块防重复（DRY）
- 敏捷 ≠ 无设计；关键设计须文档归档

## 参考

- 可扩展性 = 识别并隔离变化点
- 代码 ≠ 完整文档；深度调用链、模块依赖、业务逻辑、非功能需求须文档呈现
