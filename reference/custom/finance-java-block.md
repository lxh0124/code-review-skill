## 金额计算

- **禁止** `double`/`float` 参与金额运算或入库；DB 用 `decimal`，Java 用 `BigDecimal`
- `BigDecimal` 须用 **字符串构造**（`new BigDecimal("1.23")`），禁止 `new BigDecimal(0.1)` 或 `valueOf(double)`
- 金额比较用 `compareTo == 0`，**禁止** `equals`（`0.0` 与 `0.00` 不等）
- 统一 **分/元** 边界：对外/入库口径一致；禁止混用 `long` 分与 `BigDecimal` 元无显式转换
- 乘除须显式指定 `scale` + `RoundingMode`（如 HALF_UP）；禁止无舍入规则的 `divide`
- 出金/扣款前校验余额充足；金额 **≤0** 的支付/转账须拦截（退款/冲正除外且须单独鉴权）

## 账务与事务

- 资金变动须 **可追溯**：操作人、时间、业务单号、变动前后余额/状态、对手方
- 支付/转账/退款须保证 **事务原子性**；跨库/跨服务须有 **冲正、补偿或对账** 方案，禁止只改一半
- 支付回调、代扣、批量出款须 **幂等**（唯一业务键 + 唯一索引/状态机防重）
- 禁止在 `@Transactional` 内做 RPC、MQ、HTTP 等长耗时 I/O（易长事务锁表）
- 余额变更须 **乐观锁（version）或 `SELECT … FOR UPDATE`**，禁止 read-modify-write 无锁更新
- 禁止物理删除资金流水/账本记录；订正走冲正或补账，保留原记录

## 数据脱敏

- 日志、接口响应、异常信息 **禁止** 完整输出：银行卡号、身份证、手机号、CVV、支付密码、完整账户号
- 展示规范：银行卡保留后 4 位、手机 `158****9119`、身份证保留首尾各若干位
- **禁止** 将敏感字段写入 `debug`/`info` 或 APM/trace 自定义属性

## 安全红线

- **禁止** 硬编码密钥、证书、盐值、商户号；统一 KMS/配置中心，支持轮换
- 密码/支付 PIN **禁止明文**存库；须加盐哈希（bcrypt/Argon2 等）或 HSM
- 用户资金数据须 **水平越权** 校验（只能操作本人/本商户账户）
- 支付、绑卡、提现、代扣等写接口须 **幂等键 + 防重放**（见 ali-security 防刷）
- SQL 必须参数化；动态表名/列名须白名单（见 ali-sql-mysql）

## 加密与签名

- 传输：生产 **HTTPS/TLS 1.2+**；支付链路禁止明文敏感字段
- 存储：卡号等 PAN 须加密或 token 化；**CVV/CVC 禁止存储**（PCI）
- 国内金融场景敏感数据优先 **国密**（SM2/SM3/SM4）；AES 须足够密钥长度（如 AES-256-GCM）
- 回调/开放 API 须 **验签 + 时间戳/nonce 防重放**；禁止跳过验签分支

## 数据库（资金表）

- 业务单号、支付流水号、幂等键须有 **唯一索引**
- 资金表查询须走索引；**禁止** 无 `WHERE`/无索引全表扫描、循环单条查流水
- 大批量账务用分批 + 游标/主键范围，禁止一次加载全量到内存
- 对账字段（渠道单号、状态、金额）须可关联查询

## 合规对照（审查勾选）

| 来源 | 审查要点 |
|------|----------|
| 银保监办发〔2021〕1 号 | 权限分离、操作审计留痕 ≥6 个月、第三方组件安全评估、客户信息最小必要采集 |
| PCI-DSS | CVV 不存、PAN 加密/token、传输 TLS、密钥分级管理 |
| GDPR（涉欧业务） | 删除权/遗忘流程、最小化采集、跨境传输加密与合规评估 |
| JR/T 0171-2020 | 敏感数据分级、国密算法应用、密钥与证书生命周期管理 |
